# Bear Blend API — v1.1 Specification

**Target version:** `1.1.0`
**Estimated effort:** 1–2 days for an experienced WP developer
**Context:** v1.0.3 was built for a one-shot migration from `bearblend.com` (WooCommerce) to the Laravel rebuild at `bearblend-main.staging.humanitymedia.net`. We now need the plugin to also support (a) **ongoing nightly sync** and (b) **full-fidelity content + SEO parity** before DNS cutover. This spec closes those gaps.

All endpoints continue to use the existing Bearer token auth. No breaking changes to v1.0.3 endpoints.

---

## Motivation

Two problems this release solves:

1. **Nightly sync performance.** Today every sync walks all records from page 1 — 72K orders, 26K users, etc. A daily cron would take hours and hammer the WP instance. Adding a `modified_after` filter makes incremental sync cheap.
2. **SEO + content parity.** The WP site ranks for 2K+ keywords. All the ranking content lives in AIOSEO custom tables, static pages, menus, media alt text, and category/tag descriptions — none of which the current plugin exposes. Laravel cannot reproduce rankings without this data.

---

## New endpoints

### 1. `GET /wp-json/bb/v1/counts`

Record counts per resource type. Cheap. Lets the consumer validate sync completeness without paginating.

**Response:**
```json
{
  "customers": 26398,
  "orders": 72824,
  "products": 217,
  "product_variations": 412,
  "posts": 163,
  "pages": 21,
  "faqs": 54,
  "herbs": 10,
  "coupons": 38,
  "menus": 4,
  "media": 5812,
  "product_categories": 26,
  "product_tags": 142,
  "post_categories": 17,
  "post_tags": 89,
  "reviews": 331,
  "subscriptions": 0
}
```

**Acceptance:**
- Returns < 250ms even on cold cache.
- Count values match `wc_get_orders(['paginate' => true])->total`, `wp_count_posts()`, `wp_count_terms()`, etc. — no hand-rolled SQL.

---

### 2. `GET /wp-json/bb/v1/pages`

Static pages (About, Contact, Privacy, Terms, Shipping, Refund, etc.). Paginated like other endpoints.

**Parameters:** `page`, `per_page`, `modified_after` (ISO 8601), `include_drafts` (bool, default false), `exclude_slugs` (comma list)

**Response item shape:**
```json
{
  "id": 42,
  "title": "About Bear Blend",
  "slug": "about",
  "status": "publish",
  "content": "<html from post_content>",
  "excerpt": "...",
  "parent_id": 0,
  "menu_order": 5,
  "template": "page-landing.php",
  "date_created": "2019-03-12T14:00:00-07:00",
  "date_modified": "2024-11-01T09:15:00-07:00",
  "author_id": 1,
  "featured_image": { "id": 881, "url": "https://..." },
  "aioseo": { /* see AIOSEO shape below */ },
  "meta": { /* all wp_postmeta, flattened */ }
}
```

**Acceptance:**
- Excludes `shop`, `cart`, `checkout`, `my-account`, `thank-you`, and any page whose slug starts with `wpforms-` or `funnel-` by default.
- Returns content post-shortcode-rendering via `apply_filters('the_content', $content)` so Divi/shortcode-heavy pages are consumable.
- Honors `modified_after` (see §8 below).

### 3. `GET /wp-json/bb/v1/menus`

WordPress nav menus + menu locations. One request, all menus — this is a small dataset, no pagination needed.

**Response:**
```json
{
  "locations": {
    "primary": 12,
    "footer": 13,
    "mobile": 14
  },
  "menus": [
    {
      "id": 12,
      "name": "Main Navigation",
      "slug": "main-nav",
      "items": [
        {
          "id": 1024,
          "title": "Shop",
          "url": "https://bearblend.com/shop/",
          "target": "",
          "type": "post_type_archive",
          "object": "product",
          "object_id": 0,
          "parent": 0,
          "order": 1,
          "classes": ["menu-item", "menu-item-type-post_type_archive"],
          "description": "",
          "xfn": ""
        }
      ]
    }
  ]
}
```

**Acceptance:**
- Uses `wp_get_nav_menu_items()` — do not query `wp_posts` directly.
- Preserves parent/child ordering so the consumer can rebuild the tree in one pass.
- `url` is resolved (absolute), not just the stored meta value.

### 4. `GET /wp-json/bb/v1/media`

Media library with alt text, caption, EXIF, dimensions, file size, parent attachment.

**Parameters:** `page`, `per_page`, `modified_after`, `mime_type` (e.g. `image/jpeg`)

**Response item shape:**
```json
{
  "id": 3312,
  "title": "Bear blend gold herbal tin",
  "filename": "bear-blend-gold-herbal-tin.jpg",
  "url": "https://bearblend.com/wp-content/uploads/2020/07/bear-blend-gold-herbal-tin.jpg",
  "mime_type": "image/jpeg",
  "alt": "Bear Blend gold tin on a wooden table",
  "caption": "Our flagship blend — hand-packed in SD.",
  "description": "...",
  "width": 1600,
  "height": 1200,
  "filesize": 284912,
  "parent_post_id": 410,
  "date_created": "2020-07-15T..."
}
```

**Acceptance:**
- Alt text pulled from `_wp_attachment_image_alt` meta. This is **non-negotiable** — alt text is SEO critical.
- Dimensions + filesize pulled from `wp_get_attachment_metadata()`.
- Includes CDN URL if Cloudflare/Jetpack rewrites are active.

### 5. `GET /wp-json/bb/v1/terms`

Standalone endpoint for all taxonomies (product categories, product tags, post categories, post tags, any custom taxonomy). Returns term metadata **and descriptions** (the descriptions are ranking content).

**Parameters:** `taxonomy` (required, e.g. `product_cat`), `page`, `per_page`, `modified_after`

**Response item shape:**
```json
{
  "id": 73,
  "taxonomy": "product_cat",
  "name": "Herbal Blends",
  "slug": "herbal-blends",
  "parent_id": 0,
  "description": "Full HTML description for SEO.",
  "count": 14,
  "featured_image": { "id": 2901, "url": "https://..." },
  "aioseo": { /* see below */ },
  "meta": { /* all termmeta, flattened */ }
}
```

**Acceptance:**
- `featured_image` resolved from Yoast `thumbnail_id` termmeta OR the Woo-specific `product_cat` thumbnail ID, whichever the site uses.
- Description returned as raw HTML (do not apply `wp_trim_words`).

WooCommerce coupons — discount type, usage limits, product/category restrictions, expiry, usage counts.

**Parameters:** `page`, `per_page`, `modified_after`, `status` (default `publish`)

**Response item shape:**
```json
{
  "id": 9921,
  "code": "HEALTH20",
  "description": "...",
  "discount_type": "percent",
  "amount": "20.00",
  "date_expires": "2026-06-30T23:59:59",
  "usage_count": 412,
  "usage_limit": null,
  "usage_limit_per_user": 1,
  "individual_use": true,
  "minimum_amount": "25.00",
  "maximum_amount": "",
  "email_restrictions": [],
  "product_ids": [],
  "excluded_product_ids": [],
  "product_categories": [73, 74],
  "excluded_product_categories": [],
```

**Acceptance:**
- Uses `wc_get_coupon` / `WC_Coupon` getters — not raw post queries.

## Modifications to existing endpoints

### 8. Incremental sync: `?modified_after=` parameter on every endpoint

All existing endpoints (`/customers`, `/orders`, `/products`, `/posts`, `/herbs`, `/faqs`) **and** all new ones must accept `modified_after` as an ISO 8601 string.

**Semantics:** Return only records where `post_modified_gmt > modified_after` (for posts/CPTs), `user_registered > modified_after` OR last-meta-update > modified_after (for users — use `_last_synced_at` if you add it, otherwise fall back to `user_registered`), and WC's `date_modified_gmt` for orders/coupons.

**Acceptance:**
- Missing `modified_after` = return all records (backward compatible).
- Invalid ISO string = 400 with `bb_api_invalid_modified_after`.
- Timezone: always interpreted as UTC.
- Works in combination with `page` and `per_page`.
- For `WP_Query`, use `date_query` with `column => 'post_modified_gmt'`.
- For `wc_get_orders`, use `date_modified` arg (supports `>=` via `wc_get_orders` v3.0+).
- For `WP_User_Query`, you may need a custom `meta_query` or raw SQL — document which approach.

---

### 9. Inline AIOSEO meta on all post-based endpoints

Every response item that represents a WP post (`/products`, `/posts`, `/pages`, `/herbs`, `/faqs`, terms) must include an `aioseo` field with the full AIOSEO row for that post, or `null` if none exists.

**Shape:** See §7.

**Acceptance:**
- Single query per page of results (no N+1) — fetch `wp_aioseo_posts` rows in bulk keyed by `post_id` after the main query.
- Null-safe: don't crash if AIOSEO is deactivated.

---

### 10. `/orders` — optional `?fields=minimal` mode

For validation / delta detection, consumers want a cheap way to list order IDs + modified timestamps without the full line-items payload.

When `?fields=minimal` is passed, orders response items should only include:

```json
{
  "id": 71234,
  "status": "completed",
  "date_modified": "2026-04-21T10:15:00-07:00",
  "total": "82.50",
  "customer_id": 18021
}
```

**Acceptance:**
- Bypasses line-item / refund / fee loops entirely — pull values from the HPOS table or WC Data Store via `get_prop`.
- Returns 10x+ faster than the full response on the same page.

---

### 11. `ETag` + `Last-Modified` headers

Every paginated response should return:
- `Last-Modified` — the max `date_modified` of items in the current page.
- `ETag` — `md5($last_modified . $page . $per_page . $total)`.

When the consumer sends `If-None-Match: <etag>` and the etag matches, return **304 Not Modified** with no body.

**Acceptance:**
- Saves roundtrips during no-op nightly sync checks.
- Does not conflict with `modified_after` (they can be used together).

## Behavior & quality requirements

### Error responses
Unchanged from v1.0.3. Continue to return `WP_Error` with:
```json
{ "code": "bb_api_*", "message": "...", "data": { "status": <int> } }
```

### Authentication
Unchanged. Bearer token via `Authorization` header, stored in `bb_api_key` option, compared with `hash_equals`.

### Performance budgets
- `/counts` — < 250ms
- `/orders?fields=minimal&per_page=100` — < 1s
- `/orders?per_page=50` (full) — < 3s
- `/media?per_page=100` — < 1s
- All others with `per_page=50` — < 2s

### Versioning
Route all new endpoints under the existing `bb/v1` namespace. The `/wp-json/bb/v1/` prefix stays stable. Bump plugin version header to `1.1.0`.

### Admin settings page
Update the settings page's **Available Endpoints** table to list the new endpoints, matching the existing style.

### Tests
The plugin currently has no tests. Not required for v1.1, but please smoke-test each new endpoint from wp-cli:

```bash
wp eval 'echo rest_do_request("GET", "/bb/v1/counts")->get_data()["customers"];'
```

---

## Out of scope for v1.1

- Write endpoints (POST/PATCH/DELETE) — plugin stays read-only.
- Subscription endpoint (we can add in v1.2 once Woo Subscriptions usage is confirmed).
- Webhooks / event push — we continue to pull via cron.
- Multi-site support — single-site only.
